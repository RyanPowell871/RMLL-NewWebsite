1. ARCHITECTURE OVERVIEW
================================================================================

This project is a two-tier architecture:

  [ Browser ]  ──>  [ Static React SPA ]  ──>  [ Supabase (managed) ]
                     served from your VPS       ├── Edge Functions (Hono/Deno API server)
                     (HTML/CSS/JS files)        ├── PostgreSQL (KV store)
                                                ├── Auth (user management)
                                                ├── Storage (images, documents)
                                                └── Secrets (API keys)

  External APIs called by the Edge Functions:
    - SportzSoft API (league data): https://www.sportzsoft.com/ssRest/TeamRest.dll
    - Resend API (transactional email)
    - OpenAI API (AI features)

KEY POINT: Your VPS only needs to serve static files. All dynamic logic,
authentication, database, and storage are handled by Supabase's managed
infrastructure. This makes deployment simple and the VPS load minimal.

STEP 4: FIX IMPORT PATHS
─────────────────────────
  In the Figma Make environment, imports use bare paths like:
    import { Header } from './components/Header'

  These should work as-is in a Vite project. However:

  a. The sonner import uses a version suffix in Figma Make:
       import { toast } from 'sonner@2.0.3'
     Change ALL occurrences to:
       import { toast } from 'sonner'
     (You installed the correct version via npm already.)

  b. The react-hook-form import similarly:
       import { ... } from 'react-hook-form@7.55.0'
     Change to:
       import { ... } from 'react-hook-form'

  A quick find-and-replace across all files will handle this:
    Find:    sonner@2.0.3
    Replace: sonner

    Find:    react-hook-form@7.55.0
    Replace: react-hook-form

───────────────────────
  vite.config.ts:

    import { defineConfig } from 'vite'
    import react from '@vitejs/plugin-react'
    import tailwindcss from '@tailwindcss/vite'

    export default defineConfig({
      plugins: [react(), tailwindcss()],
      build: {
        outDir: 'dist',
        sourcemap: false,
      },
    })

──────────────────────────────────
  In your main CSS entry point (src/index.css or src/styles/globals.css),
  ensure it starts with:

    @import "tailwindcss";

  Then include the rest of your globals.css content (custom properties,
  fonts, etc.)

STEP 7: SET UP index.html
──────────────────────────
  Your index.html (in project root) should look like:

    <!DOCTYPE html>
    <html lang="en">
      <head>
        <meta charset="UTF-8" />
        <meta name="viewport" content="width=device-width, initial-scale=1.0" />
        <title>Rocky Mountain Lacrosse League</title>
        <meta name="description" content="Official website of the Rocky Mountain Lacrosse League (RMLL) - Alberta's premier box lacrosse league." />
        <link rel="icon" type="image/x-icon" href="/favicon.ico" />
      </head>
      <body>
        <div id="root"></div>
        <script type="module" src="/src/main.tsx"></script>
      </body>
    </html>

STEP 8: CREATE ENTRY POINT
───────────────────────────
  src/main.tsx:

    import React from 'react'
    import ReactDOM from 'react-dom/client'
    import App from './App'
    import './styles/globals.css'

    ReactDOM.createRoot(document.getElementById('root')!).render(
      <React.StrictMode>
        <App />
      </React.StrictMode>
    )

STEP 9: BUILD
─────────────
  npm run build

  This creates a dist/ folder containing:
    - index.html
    - assets/  (JS bundles, CSS, etc.)

  The dist/ folder is what gets uploaded to your VPS.

STEP 10: TEST LOCALLY
─────────────────────
  npm run preview

  This starts a local server serving the production build.
  Verify all pages work, API calls succeed, images load, etc.

8. VPS SERVER CONFIGURATION
================================================================================

Your InMotion VPS likely runs Apache (cPanel) or you can install Nginx.
The key requirement: ALL routes must serve index.html because this is a
Single Page Application (SPA) with client-side routing.

─────────────────────────────────────
OPTION 8A: APACHE (most likely on InMotion)
─────────────────────────────────────

  Upload the contents of dist/ to your web root, e.g.:
    /home/yourusername/public_html/

  Create or edit .htaccess in that directory:

    RewriteEngine On
    RewriteBase /

    # If the request is for an existing file or directory, serve it directly
    RewriteCond %{REQUEST_FILENAME} !-f
    RewriteCond %{REQUEST_FILENAME} !-d

    # Otherwise, serve index.html (SPA routing)
    RewriteRule ^ index.html [QSA,L]

    # Enable gzip compression
    <IfModule mod_deflate.c>
      AddOutputFilterByType DEFLATE text/html text/css application/javascript application/json image/svg+xml
    </IfModule>

    # Cache static assets aggressively (Vite adds content hashes to filenames)
    <IfModule mod_expires.c>
      ExpiresActive On
      ExpiresByType text/html "access plus 0 seconds"
      ExpiresByType text/css "access plus 1 year"
      ExpiresByType application/javascript "access plus 1 year"
      ExpiresByType image/png "access plus 1 year"
      ExpiresByType image/jpeg "access plus 1 year"
      ExpiresByType image/svg+xml "access plus 1 year"
      ExpiresByType image/webp "access plus 1 year"
      ExpiresByType font/woff2 "access plus 1 year"
    </IfModule>

    # Security headers
    <IfModule mod_headers.c>
      Header always set X-Content-Type-Options "nosniff"
      Header always set X-Frame-Options "SAMEORIGIN"
      Header always set Referrer-Policy "strict-origin-when-cross-origin"
    </IfModule>

  Ensure mod_rewrite is enabled:
    sudo a2enmod rewrite
    sudo systemctl restart apache2

─────────────────────────────────────
OPTION 8B: NGINX (if you prefer or install it)
─────────────────────────────────────

  Upload dist/ contents to, e.g., /var/www/rmll/

  Nginx config (/etc/nginx/sites-available/rockymountainlax.com):

    server {
        listen 80;
        server_name rockymountainlax.com www.rockymountainlax.com;
        return 301 https://$host$request_uri;
    }

    server {
        listen 443 ssl http2;
        server_name rockymountainlax.com www.rockymountainlax.com;

        root /var/www/rmll;
        index index.html;

        # SSL (Let's Encrypt)
        ssl_certificate /etc/letsencrypt/live/rockymountainlax.com/fullchain.pem;
        ssl_certificate_key /etc/letsencrypt/live/rockymountainlax.com/privkey.pem;

        # SPA routing — serve index.html for all non-file routes
        location / {
            try_files $uri $uri/ /index.html;
        }

        # Cache static assets (Vite hashes filenames, safe to cache forever)
        location /assets/ {
            expires 1y;
            add_header Cache-Control "public, immutable";
        }

        # Gzip
        gzip on;
        gzip_types text/css application/javascript application/json image/svg+xml;
        gzip_min_length 1000;

        # Security headers
        add_header X-Content-Type-Options "nosniff" always;
        add_header X-Frame-Options "SAMEORIGIN" always;
        add_header Referrer-Policy "strict-origin-when-cross-origin" always;
    }

  Enable and restart:
    sudo ln -s /etc/nginx/sites-available/rockymountainlax.com /etc/nginx/sites-enabled/
    sudo nginx -t
    sudo systemctl restart nginx

9. WORDPRESS MIGRATION STRATEGY
================================================================================

You have several options for transitioning from WordPress:

─────────────────────────────────────
STRATEGY A: CLEAN CUTOVER (Simplest)
─────────────────────────────────────
  1. Back up current WordPress site (files + database)
  2. Set up the new React site in a test location first:
       - Upload to a subdomain: test.rockymountainlax.com
       - Or use a temporary domain to verify everything works
  3. Once verified, replace WordPress files with React build files
  4. Update .htaccess for SPA routing
  5. WordPress is fully replaced

  TIMELINE: 1-2 hours of downtime during cutover

─────────────────────────────────────
STRATEGY B: PARALLEL RUN (Safer)
─────────────────────────────────────
  1. Move WordPress to a subdomain:
       - old.rockymountainlax.com or wp.rockymountainlax.com
       - Update WordPress Settings > General > WordPress Address
  2. Deploy React app to the main domain
  3. Keep WordPress running for reference / content migration
  4. Shut down WordPress when comfortable

  TIMELINE: Zero downtime

─────────────────────────────────────
STRATEGY C: SUBDIRECTORY (Advanced, not recommended)
─────────────────────────────────────
  Run both side by side with Apache routing rules. Not recommended due
  to complexity and potential routing conflicts with the SPA.

─────────────────────────────────────
WORDPRESS CONTENT TO MIGRATE:
─────────────────────────────────────
  Before shutting down WordPress, ensure you've migrated:
  [ ] Any news/blog posts → CMS News Manager
  [ ] Uploaded images → CMS Image Manager (Supabase Storage)
  [ ] Documents/PDFs → CMS Document Manager (Supabase Storage)
  [ ] Contact form recipients → CMS Settings
  [ ] Any SEO metadata/redirects you want to preserve

11. SUPABASE PRODUCTION HARDENING
================================================================================

Your Supabase project is already set up and running. Here's what to verify
and harden for production:

PLAN:
  - The Free plan limits: 500MB database, 1GB storage, 50K Edge Function
    invocations/month, 2 million auth requests/month
  - For a league website with moderate traffic, the Pro plan ($25/month)
    is recommended for: higher limits, daily backups, no pausing after
    1 week of inactivity, email support
  - Upgrade at: https://supabase.com/dashboard/project/nkfbehspyjookipapdbp/settings/billing

EDGE FUNCTIONS:
  The Edge Functions in /supabase/functions/server/ are already deployed
  and running. If you need to redeploy after changes:

  1. Install Supabase CLI:
       npm install -g supabase
  2. Login:
       supabase login
  3. Link your project:
       supabase link --project-ref nkfbehspyjookipapdbp
  4. Deploy the edge function:
       supabase functions deploy server --no-verify-jwt

  NOTE: The edge function is named "server" and contains all routes via
  the Hono framework. The --no-verify-jwt flag is needed because the
  function handles its own auth verification internally.

SECRETS (already configured):
  Verify all secrets are set in the Supabase dashboard under
  Project Settings > Edge Functions > Secrets:

  - SPORTZSOFT_API_KEY    (SportzSoft league data API)
  - RESEND_API_KEY        (Transactional email via Resend)
  - OPENAI_API_KEY        (AI features)

  These are automatically available as:
  - SUPABASE_URL          (auto-set by Supabase)
  - SUPABASE_ANON_KEY     (auto-set by Supabase)
  - SUPABASE_SERVICE_ROLE_KEY (auto-set by Supabase)
  - SUPABASE_DB_URL       (auto-set by Supabase)

ROW LEVEL SECURITY (RLS):
  The kv_store_9a1ba23f table should have RLS enabled with appropriate
  policies. Check the Supabase dashboard > Table Editor > kv_store_9a1ba23f
  > RLS Policies. Since all database access goes through Edge Functions
  (which use the service role key), RLS can be permissive for the service
  role, but should block direct client access.

STORAGE:
  The buckets make-9a1ba23f-images and make-9a1ba23f-documents are created
  automatically by the server on startup. Verify storage policies in the
  dashboard allow public read access for images but restrict uploads to
  authenticated requests through the Edge Function.

BACKUPS:
  - Pro plan includes daily backups with 7-day retention
  - For additional safety, periodically export KV data via the CMS
  - Document storage files are in Supabase Storage (replicated)

15. ROLLBACK PLAN
================================================================================

  If something goes wrong after deployment:

  FRONTEND ROLLBACK:
    Keep the previous dist/ folder as a backup before each deploy.
    To rollback: replace current files with the backup dist/ folder.

    # Before deploying:
    ssh user@vps "cp -r /home/user/public_html /home/user/public_html_backup_$(date +%Y%m%d)"

    # To rollback:
    ssh user@vps "rm -rf /home/user/public_html && mv /home/user/public_html_backup_YYYYMMDD /home/user/public_html"

  WORDPRESS ROLLBACK:
    If you kept the WordPress backup (you should!), you can restore it by:
    1. Re-uploading WordPress files to public_html
    2. Restoring the MySQL database
    3. Removing the SPA .htaccess rules

  SUPABASE ROLLBACK:
    Edge Functions can be redeployed to any previous version.
    Database changes through KV are additive (set/delete individual keys).
    Pro plan includes point-in-time recovery.

16. TROUBLESHOOTING
================================================================================

  PROBLEM: Pages return 404 when accessing URLs directly
  CAUSE:   .htaccess SPA rewrite rules not working
  FIX:     Ensure mod_rewrite is enabled and AllowOverride All is set
           in Apache config for your directory

  PROBLEM: API calls fail with CORS errors
  CAUSE:   Supabase Edge Functions not returning CORS headers
  FIX:     The Hono server already has CORS configured with origin: "*"
           This should work. Check browser console for the actual error.

  PROBLEM: SportzSoft data not loading
  CAUSE:   API key not configured or expired
  FIX:     Check Supabase secrets, verify key at SportzSoft admin panel

  PROBLEM: Images/documents not loading
  CAUSE:   Supabase Storage bucket permissions
  FIX:     Check bucket policies in Supabase dashboard; images bucket
           should allow public read access

  PROBLEM: CMS login fails
  CAUSE:   Auth configuration or user not created
  FIX:     Verify user exists in Supabase Auth dashboard; check KV for
           user profile with admin role

  PROBLEM: Blank page on load
  CAUSE:   JavaScript bundle not loading; check browser console
  FIX:     Verify index.html and assets/ are in the correct directory;
           check that the script src path is correct

  PROBLEM: Fonts not loading (Athletic Block, Bebas Neue, Montserrat)
  CAUSE:   External font CDN blocked or slow
  FIX:     These load from Google Fonts and cdnfonts.com via CSS @import;
           consider self-hosting fonts for better performance

  PROBLEM: Build fails locally
  CAUSE:   Missing dependencies or TypeScript errors
  FIX:     Run npm install, check for any Figma Make-specific imports
           that need adjustment (sonner@2.0.3 → sonner, etc.)
